Known Problems


_____________________________________________________________________________________________

Description: G compset, T62_t12 resolution PE-layout not optimal 
Date       : 03/31/2010
Contact    : fischer 
Information: 

An optimal PE-layout is not produced, suggest the use of an alternative PE-layout, 
or increase the wall clock limit to 8 hours.

_____________________________________________________________________________________________

Description: C compset, T62_t12 resolution timestepping instability
Date       : 03/31/2010
Contact    : njn01
Information:

There is an initial timestepping instability in POP2 that results in the the model
issuing the error message "ERROR: k.e. > 100" and shutting down. The solution to this
problem is to reduce the POP2 timestep size (increase DT_COUNT). 

We are continuing to test this configuration to determine a recommendation for an optimal
initial timestep size. We know that an initial value of the ocean-model timestep-control
parameter DT_COUNT = 400 is successful for a six-day startup run. Most likely, the user
will be able to decrease DT_COUNT (increase the timestep size) following an initial startup
period.

Check back later for more information.

_____________________________________________________________________________________________

Description: Threaded active-ocean rake decomposition failure
Date       : 03/31/2010
Contact    : njn01
Information:

Occasionally, when running CCSM4 POP2 with OpenMP and using certain processor configurations, 
the POP_Distribution rake algorithm fails when clinic_distribution_type is set to its default 
value, balanced. 

Under these circumstances, the user should set clinic_distribution_type = cartesian instead.

_____________________________________________________________________________________________
Description: incorrect create_newcase messages regarding AMIP compset
Date       : 04/02/2010
Contact    : kauff
Information:

create_newcase issues the message:
  * If the user is interested in running a "stand-alone"
  * component configuration, the following model
  * configurations have been validated scientifically and
  * have associated diagnostic output as part of the release.
  *
  *  0.9x1.25_gx1v6  F_AMIP_1DEG
  *  1.9x2.5_gx1v6   F_AMIP_2DEG

This is incorrect.  It should say
  *  0.9x1.25_0.9x1.25 F_AMIP
  *  1.9x2.5_1.9x2.5   F_AMIP

Also, if one tries to create an AMIP case with an unsupported resolution
you get the error message...
   "ERROR: Default resoluton independent AMIP is INVALID"

Description: half degree surface dataset for clm repeats the path twice
Date       : 04/29/2010
Contact    : erik
Information:

When using an 1850 compset with CLM, the path to the surface dataset is
repeated twice. So modify your namelist in Buildconf/clm.buildnml.csh to change
the path for "lnd/clm2/surfdata/lnd/clm2/surfdata" to "lnd/clm2/surfdata".
